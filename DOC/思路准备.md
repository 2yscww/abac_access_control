## 思路准备

#### ABAC影响主要因素

```
Subject (S): 用户的职级、部门、项目组、信用分。

Resource (R): 文档的密级、所属部门、创建者、当前状态。

Action (A): 读取、修改、导出、审批。

Environment (E): 访问时间、IP段、风险等级、地理位置。

```

<br/>

#### ABAC组成

| 缩写     | 全称 |形象比喻    |在医疗平台中的职责|
| ----------- | ----------- |------|-----|
| PEP      | Policy Enforcement Point (策略执行点)       |门卫/保安|拦截医生的 API 请求。它不决定能不能进，只负责“拦截”和“执行结果”。|
| PDP   | Policy Decision Point (策略决策点)        |法官|权限引擎的核心。它拿到请求，根据规则算出“允许”还是“拒绝”。|
|PIP|Policy Information Point (策略信息点)|证人/档案室|提供额外信息。例如：PDP 发现规则里有“科室对比”，PIP 就去数据库查该医生到底属于哪个科室。|
|PAP|Policy Administration Point (策略管理点)|立法机构|管理员写规则的地方。比如规定“非工作时间不能看 L4 级数据”。|

<br/>


#### 下达决心

| 模块                     | 是否毕业设计级 / 评价说明 |
|--------------------------|----------------------------|
| ABAC 核心模型             | ✅ 毕设核心内容，体现权限模型设计能力 |
| 多阶段权限控制             | ✅ 体现复杂业务流程下的动态权限变化 |
| 策略冲突裁决机制           | ✅（加分项）体现复杂规则下的决策与治理能力 |
| 缓存与性能优化             | ✅（工程能力）体现系统性能与可扩展性设计 |
| 安全审计模块               | ✅（标题对齐）支撑安全性与可追溯性分析 |
| 临时授权 / 例外授权机制     | ⭐（优秀毕设）体现真实业务中的弹性权限管理 |
| 可视化的策略树编辑（可选） | ⭐（展示型加分）提升系统可用性与策略可理解性 |

#### 业务场景总体描述

| 项目项         | 说明                                                                 |
|----------------|----------------------------------------------------------------------|
| 场景名称       | 面向多项目协作的企业敏感文档访问控制与安全审计系统                   |
| 场景类型       | 企业级内容管理（ECM）子场景                                         |
| 适用对象       | 中大型企业内部员工、项目成员、外包人员                               |
| 核心问题       | 多部门、多项目协作下，敏感文档访问权限难以精细控制                   |
| 解决目标       | 通过 ABAC 实现基于多属性的动态访问控制与安全审计                     |
| 权限复杂性来源 | 用户属性、资源属性、环境属性交叉决策                                 |
| 模型选择原因   | RBAC 难以覆盖跨项目、密级、环境等复杂授权需求                         |

#### 想定场景的业务流程

```
立项 → 需求设计 → 研发实现 → 测试验证 → 上线交付 → 归档
```


#### 参与部门与职责划分
| 部门       | 部门职责说明                                           |
|------------|----------------------------------------------------------|
| 产品部     | 负责需求调研、需求文档编写、需求评审                     |
| 研发部     | 负责系统设计、代码开发、技术文档编写                     |
| 测试部     | 负责测试用例设计、功能/性能/安全测试                     |
| 运维部     | 负责部署方案、上线运维、运行监控                         |
| 行政/管理层| 负责项目审批、进度监管、审计与合规                       |


<br/>

#### 项目阶段访问控制矩阵（核心业务规则）
| 项目阶段     | 可访问部门         | 禁止访问部门           | 业务说明 |
|--------------|--------------------|------------------------|----------|
| 立项阶段     | 产品部、管理层     | 研发部、测试部、运维部 | 仅讨论可行性与预算，技术人员不介入 |
| 需求设计阶段 | 产品部、研发部     | 测试部、运维部         | 需求尚未冻结，避免过早扩散 |
| 研发实现阶段 | 研发部、产品部     | 测试部、运维部         | 核心代码与设计文档高度敏感 |
| 测试验证阶段 | 测试部、研发部     | 运维部                 | 防止未上线系统被提前部署 |
| 上线交付阶段 | 运维部、研发部     | —                      | 系统部署与稳定性保障 |
| 归档阶段     | 管理层（只读）     | 其他所有部门           | 项目结束，仅用于审计 |

```
优先级：安全策略 > 项目策略 > 角色属性
```
